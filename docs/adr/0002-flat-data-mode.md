# DATA_MODE 保持单一扁平设置，不拆分独立的「数据源」轴

## 决定

数据模式只用一个设置 `DATA_MODE`，四选一：`jnb | qcore | akshare | websearch`。
**不**再额外引入 `DATA_SOURCE` 设置去单独表达「数据从哪来」。

## 为什么

- **qcore 与 akshare 同源**：两者是同一份数据（akshare 前复权日线）的两种取法 —— qcore 是本地缓存（快、需本地数据湖），akshare 是现拉（慢、免配置）。它们不是两个不同的数据提供商，只是「同一份数据的两种拿法」。
- **Tushare 已休眠且无盘中接口**：ADR-0001 里「留作他日接盘中实时」只是一句备注，代码里并没有任何盘中实时的实现，也没有 `DATA_SOURCE` 的任何痕迹。
- **真实数据源只有一族**：既然能拿数据的「源」实际上只有 akshare 这一族（qcore 是它的缓存，Tushare 睡着了），那么为这「一轴半」的来源专门拆出「模式 × 数据源」两个正交设置，属于过度设计 —— 只会平白给 `data_sync` / `setup_wizard` / 测试增加一层永久的维护面，却没有对应的真实需求。

## 什么时候该推翻这个决定

当出现一个**语义真正不同**的数据源（例如字段、单位、频率都和现有日线不一样的盘中实时源），并且**真的接上了接口、有消费方**时 —— 那时再把「数据源」独立成轴才划算。在那之前，扁平枚举就是更耐用的设计。
